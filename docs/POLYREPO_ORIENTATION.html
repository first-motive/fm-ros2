How To Use The Repo Day To Day

Normal entrypoints

  • ./run.sh is the front door.
  • ./scripts/sim.sh launches one robot/backend directly.
  • ./scripts/teleop.sh adds Servo + an input path.
  • ./scripts/view-robot.sh is for robot description viewing only.

Important directories

  • src/ contains the actual layer repos.
  • external/ contains vendored robot and learning dependencies.
  • scripts/ contains the operational helpers you actually run.
  • docs/ contains the cross-repo orientation docs.
Make code changes in the smallest owning repo possible. Use fm-ros2 for orchestration and runbooks, not for new robot/sim/training feature code unless the change truly belongs there.

Exact Steps: Run a Simulation on a Mac

The validated local Mac path in this workspace is SO101 in MuJoCo with Foxglove on top. This is the daily-driver simulation path for macOS.

  1. Open a terminal and go to the orchestrator root.
cd /Users/retief/Documents/git-work/fm-polyrepo/fm-ros2
  1. Make sure OrbStack is installed and running, and Foxglove Studio is installed on the Mac.
  2. If you need a fresh one-time build, run:
docker compose -f docker/compose.yaml -f docker/compose.macos.yaml build
docker compose -f docker/compose.yaml -f docker/compose.macos.yaml run --rm fm_ros2 ./scripts/import-externals.sh
docker compose -f docker/compose.yaml -f docker/compose.macos.yaml run --rm fm_ros2 colcon build --symlink-install
  1. Launch the validated SO101 scene:
./scripts/sim.sh --robot so101 --backend mujoco --task-env pick_place
  1. Open Foxglove Studio and connect to ws://localhost:8765.
  2. In a 3D panel, add:
/tf
/tf_static
/robot_description
/joint_states
/task_env_markers
  1. Set the fixed frame to base_link.
  2. You should now see the SO101 arm, the table/props, and live joint updates.

Other verified task environments:

./scripts/sim.sh --robot so101 --backend mujoco --task-env table_reach
./scripts/sim.sh --robot so101 --backend mujoco --task-env bin_sort

Quick verification from another terminal:

cd /Users/retief/Documents/git-work/fm-polyrepo/fm-ros2
docker compose -f docker/compose.yaml -f docker/compose.macos.yaml exec -T fm_ros2 /ros_entrypoint.sh ros2 topic list | sort
docker compose -f docker/compose.yaml -f docker/compose.macos.yaml exec -T fm_ros2 /ros_entrypoint.sh ros2 control list_controllers

Stop the stack:

docker compose -f docker/compose.yaml -f docker/compose.macos.yaml down
On macOS, the validated sim backends are mock and mujoco. This path is for dev, build, sim, and dataset work. It is not the GPU or real hardware path.

Exact Steps: Run Simulation on Linux

Linux is the full native path for GPU, real hardware, and Linux-only sim backends. The setup script checks Docker, warns if NVIDIA container runtime is missing, opens local X11 access, imports externals, and builds the Linux image.

  1. Go to the orchestrator root:
cd /path/to/fm-polyrepo/fm-ros2
  1. Run the Linux setup helper:
./scripts/setup-linux.sh
  1. Build the workspace in the Linux overlay:
docker compose -f docker/compose.yaml -f docker/compose.linux.yaml \
  run --rm fm_ros2 colcon build --symlink-install
  1. Launch through the front door if you want the menu flow:
./run.sh --linux
  1. Or launch a Linux-native simulator directly.
./scripts/sim.sh --robot so101 --backend gazebo --task-env pick_place
# or
./scripts/sim.sh --robot so101 --backend isaac --task-env pick_place

Important: in the checked-in script logic, sim.sh maps mock and mujoco to the macOS compose overlay, while gazebo, isaac, and real map to the Linux overlay. So the Linux-native backend choices are the GPU path.

Bring the shared stack down when finished:

docker compose -f docker/compose.yaml -f docker/compose.linux.yaml down
If you want the same operator experience as on the Mac, use ./run.sh --linux and connect Foxglove to ws://localhost:8765. If you want GPU-backed simulation or hardware, stay on Linux.

Exact Steps: Do VLA Training

In this workspace, the concrete training path lives in src/fm-policy. The main training package is fm_policy_train, and the best deeper runbook is src/fm-policy/docs/libero_5090_pipeline_runbook.md.

The operational mental model is: data and episodes live in fm-data, training lives in fm-policy, and fm-learning groups them. For actual GPU training, use Linux.

  1. Go to the training repo inside this workspace:
cd /Users/retief/Documents/git-work/fm-polyrepo/fm-ros2/src/fm-policy
  1. Set up the reference LIBERO environment if you want compatibility checks:
fm_policy_train/scripts/setup_libero_env.sh --profile ref_py38
  1. Set up the real RTX 5090 training environment on Linux for actual training runs:
fm_policy_train/scripts/setup_libero_env.sh --profile rtx5090
  1. Download the LIBERO spatial dataset:
conda run -n libero_ref_py38 python \
  fm_policy_train/scripts/download_libero_data.py \
  --suite libero_spatial
  1. Run a small smoke pass first:
conda run -n libero_5090_cuda python \
  fm_policy_train/scripts/run_libero_experiments.py \
  --profile rtx5090 \
  --alignment-mode forward_probe \
  --run-budget smoke \
  --seeds 42 \
  --output-root runs/libero_smoke
  1. Then run the benchmark-style pass:
conda run -n libero_5090_cuda python \
  fm_policy_train/scripts/run_libero_experiments.py \
  --profile rtx5090 \
  --alignment-mode lifelong_benchmark \
  --run-budget benchmark \
  --seeds 42,1337,90210 \
  --output-root runs/libero_5090_batch96_workers4
  1. Summarize the finished run:
conda run -n libero_5090_cuda python \
  fm_policy_train/scripts/summarize_libero_run.py \
  --run-root runs/libero_5090_batch96_workers4/libero_spatial/bc_transformer_policy/rtx5090

What the current training scripts assume:

  • external/libero/LIBERO is where the LIBERO repo is cloned and installed editable.
  • datasets/libero_spatial holds the downloaded HDF5 datasets.
  • runs/... holds training outputs and checkpoints.
  • fm_policy_train/config/libero_hardware.yaml holds the hardware knobs.
For this repo, “VLA training” practically means starting in fm-policy and following the LIBERO pipeline. Read src/fm-policy/fm_policy_train/README.md first, then src/fm-policy/docs/libero_5090_pipeline_runbook.md for the deeper operator playbook.
